# Payments System

This is a toy payments system where I show how to handle most common transaction operations

## System design

### Only Deposits can be disputed

In this system, only deposit transactions can be disputed. 
This is because the dispute is considered to be only in the client that got the money (Deposit)
and not in the client that sent the money (Withdrawal)

### Type-state pattern

The system uses the type-state pattern to ensure that only valid operations can be performed on transactions based on their current state.
For example, a transaction that is in the "chargeback" state cannot be disputed again.

### Rounding Strategy

Monetary values are rounded to **4 decimal places** using `RoundingStrategy::MidpointNearestEven` (banker's rounding).

This policy is implemented in `MoneyAmount::new` (`src/models.rs`) and is used to reduce cumulative rounding bias across many transactions.

### Deserialization error handling

When deserializing transactions from CSV, any errors (e.g., unsupported transaction types) 
will stop the processing of the entire file with no updates applied.

This ensures that only fully valid transaction files are processed, and output is only generated with accurate results.

On a server application with a real database keeping track of all the transactions and super large csv files, 
I would implement a more robust strategy where we make a partial process of the file and log the invalid row.
This will allow the system to resume processing saving time and resources while keep the data consistent and as up to date as possible.

### Dispute resolution

The system handles disputes by allowing clients to dispute transactions, which locks the disputed amount until the dispute is resolved.

The resolution of a dispute is final which means a second dispute cannot be opened for the same transaction.
